The zKB API is for developers and enthusiasts who want to integrate zKB into their website or application. Usage is provided under the following rules: - Do not hammer the server with API requests. - Follow the cachetime indicated in the expires header (You can use the cachedUntil value in the XML if you must, but the expires header is more accurate). - If you must do several requests in a row, wait a few seconds between each. - Please send 'Accept-Encoding: gzip' as a header, to help save bandwidth. - Please send 'User-Agent: ' as a header, describing who you are, so we don't ban legitimate users. Also you should keep these things in mind: - Remember to look at the X-Time-Between-Req header, it lists how many seconds you have to wait between requests. - If you get an Error 403, look at the Retry-After header, for when you can do another request. - All IDs used with the API are CCP IDs. - The API will maximum deliver 200 mails back at a time. - If the /limit/ modifier is used then /page/ will be unavailable. - startTime and endTime is a datetime timestamp, in the format: YmdHi (Example: 2012-11-25 19:00 = 201211251900). - W-space and solo can be combined with kills and losses. - api-only will return only CCP API Verified kills. - To get combined kills and losses, don't use the kills or losses modifiers. - Up to 10 IDs can be fetched at the same time, by seperating them with a , (comma). - pastSeconds returns only kills that have happened in the past x seconds. - pastSeconds can maximum go upto 7 days (604800 seconds) - Default return is JSON, you can get XML if you pass the XML parameter in the URL. - Remember to finish with a / (forward slash). - All modifiers can be combined in any order. - You must pass atleast two modifiers. (w-space / solo or an xID is needed) - Negative killIDs denote manually posted mails, positive killIDs denote API posted mails. - JSONP is available, but untested. - page requests > 10 are only allowed for characterID, corporationID, and allianceID Defaults: - Default sort is desc (latest to oldest). - Default page is 1, second page is 2, third is 3, etc. - Default is json Fetch type modifiers - kills - losses - w-space - solo Page / time / limit modifiers: - limit/numkills/ - page/pagenumber/ - startTime/datetime/ - endTime/datetime/ - year/yearnumber/ - month/monthnumber/ - week/weeknumber/ - beforeKillID/id/ - afterKillID/id/ - pastSeconds/seconds/ Order modifiers (Not working atm): - orderDirection/asc/ - orderDirection/desc/ Fetch modifiers: - characterID - corporationID - allianceID - factionID - shipTypeID - groupID - solarSystemID - regionID Less Information - no-items - no-attackers - api-only Other Format - xml Examples: -- https://zkillboard.com/api/kills/w-space/ -- https://zkillboard.com/api/kills/w-space/xml/ -- https://zkillboard.com/api/losses/solo/ -- https://zkillboard.com/api/losses/solo/xml/ -- https://zkillboard.com/api/kills/characterID/268946627/ -- https://zkillboard.com/api/kills/characterID/268946627/xml/ -- https://zkillboard.com/api/solo/kills/characterID/268946627/ -- https://zkillboard.com/api/kills/regionID/10000002/ -- https://zkillboard.com/api/w-space/corporationID/98076299/ -- https://zkillboard.com/api/w-space/corporationID/98076299/page/2/ -- https://zkillboard.com/api/kills/shipID/644,638/orderDirection/asc/ Stats API For those people wishing to pull general statistics about character/corporation/alliance or faction there is an additional stats API. This API will not accept any of the modifiers used above. Allowed Types - characterID - corporationID - allianceID - factionID - shipID - groupID - systemID - regionID Examples: -- https://zkillboard.com/api/stats/character/268946627/ -- https://zkillboard.com/api/stats/corporation/98076299/xml/ -- https://zkillboard.com/api/stats/alliance/386292982/json/ Please Note: If you aren't using the api-only modifier, then it is possible to pull kills that include a negative killID. Negative killIDs denote manual mails. Those manual mails could later get verified by the API, and get a positive killID (API mails). This means you could potentially pull the same kill twice. It is up to you to detect this.
